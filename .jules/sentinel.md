## 2024-05-24 - [CRITICAL] Insecure Deserialization in CacheManager
**Vulnerability:** The `CacheManager` used the `pickle` module (`pickle.load` and `pickle.dump`) to serialize and deserialize cached API responses from the disk. This allowed for arbitrary code execution if an attacker could modify the `.cache` files on disk, as `pickle` can instantiate arbitrary objects and run malicious code during deserialization.
**Learning:** `pickle` is inherently unsafe for deserializing data from untrusted or potentially modified sources (like files on disk in an exposed environment). Even if the application wrote the data, it's safer to use a format that doesn't allow object instantiation. Also, when changing serialization formats (e.g., from `pickle` to `json`), it is critical to change the file extension (e.g., `.cache` to `.json_cache`) to prevent the application from crashing when trying to parse legacy binary cache files with a text parser (like `json`), which raises unhandled `UnicodeDecodeError`.
**Prevention:** Always use safe serialization formats like `json` instead of `pickle` for caching data, especially when the data consists of simple structures (dicts, lists, strings) typical of API responses.
